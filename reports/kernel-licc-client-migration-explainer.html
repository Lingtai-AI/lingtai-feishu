lingtai-feishu: prefer kernel LICC client

Migration PR prepared for the first-party MCP LICC interface cleanup.

Conclusion

Ready for review. This PR keeps the addon-compatible lingtai_feishu.licc.push_inbox_event import path, but delegates to the canonical kernel helper lingtai.core.mcp.licc.push_inbox_event whenever the host LingTai runtime provides it.

What changed

  • Replaced the purely vendored LICC writer with a compatibility wrapper.
  • Added a local fallback so standalone development and older runtime environments continue to work before the kernel PR/release is available.
  • Updated README wording so the kernel helper is documented as canonical and this repo's licc.py is documented as a compatibility wrapper.
  • Added focused tests for kernel delegation and fallback event writing.

Risk controls

  • No runtime configuration changes.
  • No hard lingtai package dependency added, avoiding a packaging dependency cycle while the kernel package still depends on first-party MCP addons.
  • If the canonical helper is unavailable, behavior falls back to the same LICC v1 filesystem contract.
  • No merge should happen tonight; Jason explicitly asked to prepare these PRs for tomorrow's review/migration window.

Validation

Run locally before opening PR:

  • python -m pytest tests/test_licc_wrapper.py -q
  • python -m compileall src/lingtai_feishu/licc.py tests/test_licc_wrapper.py
  • git diff --check

Dependency

Operationally depends on kernel PR #229

The addon can merge independently because of the fallback, but it will only use the canonical kernel helper when running with a LingTai kernel version that includes lingtai.core.mcp.licc.
